1. Introduction

The purpose of this document is to specify the ISI messages for GRANITE private interface. There are also common messages which must be supported by every server (see reference /4/).

More general guidelines about ISI server specification can be found from references /2/ and /3/.

2. Message Header

Every ISI message contains a message header. The header is based on the Phonet message header plus some extra fields specific to ISI. More details concerning the ISI message header and how to build and use an ISI message can be found in ISI General Information document (see reference /3/).
The ISI message specifications do not include the complete header information in each message, as this is unnecessary. The ISI message specifications refer only to the following field from the message header.
 · Resource - This describes the function or resource (and therefore the server) that the message belongs to, e.g. PN_SECURITY, PN_CALL etc.

Message specifications use the following format.

Size in bytes Parameter Description
3 Message header info As in the PhoNet header
1 Resource PN_PRIVATE
4 Message header info As in the PhoNet header
1 Transaction ID  
1 Message ID  
2 Private ID  
1 Some message specific data  
1 Some more message specific data  
1 Even more message specific data

3.3.  GRN_TEST_REQ

Request message. Corresponding response message(s):
 GRN_TEST_RESP

Test step is encoded in WBXML and put in a GRN_WBXML_REQ_INFO subblock. If the test step is too long for a single message, it can be split. Each message except the last one must have TRUE in the Continue field. A response (GRN_TEST_RESP) is sent only after the last request message except in case of error. The server may respond with a notification (GRN_TEST_NTF) to each message.
This message also means that the results of the previous test step will not be queried anymore.

Size in bytes Parameter Description
1 Transaction ID  
1 Message ID GRN_TEST_REQ (= 0x02)
2 Private ID PN_PRIVATE_GRANITE (= 0xFF0D)
1 Continue Possible values are TRUE and FALSE, which are defined in the file global.h

Default value: FALSE

TRUE if message is incomplete and continued in the next message. FALSE in the last messsage. 
2 Filler  
1 Number of Sub Blocks Reference name: N

Possible value: 1
? Sub blocks. If N > 0 GRN_WBXML_REQ_INFO

3.4.  GRN_TEST_RESP

Response message to request message(s):
 GRN_TEST_REQ

This message is sent when the requested test step is executed or an error condition is detected. Error details are given in a subblock if there was a syntax error in the test request. Otherwise there is no subblock. The result of the test is not in the message but can be queried with GRN_RESULT_REQ messages.

Size in bytes Parameter Description
1 Transaction ID  
1 Message ID GRN_TEST_RESP (= 0x03)
2 Private ID PN_PRIVATE_GRANITE (= 0xFF0D)
1 Continue Possible values are TRUE and FALSE, which are defined in the file global.h

Default value: FALSE

TRUE if message is incomplete and continued in the next message. FALSE in the last messsage. 
1 Response Possible values are OK and FAIL, which are defined in the file global.h 
1 Cause Constant: GRN_CAUSES 
1 Number of Sub Blocks Reference name: N

Possible values: 0, 1
? Sub blocks. If N > 0 GRN_PARSE_ERROR_INFO

3.5.  GRN_TEST_NTF

These notifications may be sent as preliminary responses to GRN_TEST_REQ requests. Server need not send all notifications. Future versions of server may send other notifications. Unidentified notifications (content of Notify field not any of values in GRN_NOTIFICATIONS) can be ignored.
Notifications as specified here have no subblocks. Future versions of GRANITE server may put subblocks to notifications. Unexpected subblocks can be ignored.

1 Notify Constant: GRN_NOTIFICATIONS 
1 Filler

? Sub blocks. If N > 0

3.13.  GRN_RESULT_REQ

Request message. Corresponding response message(s):
 GRN_RESULT_RESP

After execution of test case the results can be queried with GRN_RESULT_REQ messages. Result must be queried before any of GRN_TEST_REQ or GRN_CLEAR_REQ or GRN_TERM_REQ message, for any one of these clears the result in the server. The result or a part of it is returned in a GRN_RESULT_RESP message. Several requests may be needed to get all of the result.

Size in bytes Parameter Description
1 Transaction ID  
1 Message ID GRN_RESULT_REQ (= 0x0F)
2 Private ID PN_PRIVATE_GRANITE (= 0xFF0D)
1 Continue Possible values are TRUE and FALSE, which are defined in the file global.h

Default value: FALSE

This message cannot be continued 
2 Filler  
1 Number of Sub Blocks Reference name: N

Possible values: 0, 1
? Sub blocks. If N > 0 GRN_INDEX_INFO

3.14.  GRN_RESULT_RESP

Response message to request message(s):
 GRN_RESULT_REQ

Result of last test or a part of it. The GRN_INDEX_INFO subblock (if present) identifies the part. The GRN_WBXML_RESP_INFO subblock (if present) contains the result in WBXML encoding. The latter or both subblocks may be absent if there is no result or the result is so short that the requested part does not exist.

Size in bytes Parameter Description
1 Transaction ID  
1 Message ID GRN_RESULT_RESP (= 0x10)
2 Private ID PN_PRIVATE_GRANITE (= 0xFF0D)
1 Continue Possible values are TRUE and FALSE, which are defined in the file global.h

Default value: FALSE

This message cannot be continued 
1 Response Possible values are OK and FAIL, which are defined in the file global.h 
1 Cause Constant: GRN_CAUSES 
1 Number of Sub Blocks Reference name: N

Value range: 0 to 2
? Sub blocks. If N > 0 GRN_INDEX_INFO
GRN_WBXML_RESP_INFO

4.3.  GRN_WBXML_REQ_INFO

The length of this subblock can be anything (whithin limits). Therefore this subblock should be the last, in order to avoid bad alignment in next subblocks.
Test step is encoded in WBXML. If a test step is too long for a message, the WBXML encoded test step may be split at any point and parts be sent in separate messages.
There are various constrains on the WBXML encoding. The global tokens have their normal encodings that is shown in the table GRN_WBXML_GLOBALS, but most of them (marked "Not used") shall not be used. The WBXML header bytes must have values from tables GRN_WBXML_VERSIONS, GRN_WBXML_PUBLIC_IDS, and GRN_WBXML_CHARSETS. The string table shall not be used (so the length of the string table is zero, and shall be encoded as a single zero byte. All XML tags shall be encoded by codes shown in the table GRN_WBXML_TAGS. Tags not shown in the table shall not be used. First byte of an attribute identifies the label of the attribute and shall be encoded with a single byte as shown in table GRN_WBXML_ATTR_LABELS. The first byte shall be followed by the encoded attribute value (of the appropriate type). The value shall not be encoded as a string unless it really is a string value. String values start with the GRN_WBXML_STR_I byte, followed by the string in UTF-8 encoding, and end with a zero byte. Integer values shall be encoded with GRN_WBXML_EXT_T_0 byte followed by mb_uint32 encoding of the value. However, for small values can be encoded with a single byte. Single byte encodings are shown in the table GRN_WBXML_ATTR_VALUES. These encodings shall be used for attributes with a limited set of valid values. More than one byte from the table shall not be used except for items in capture tag. Attribute value may be omitted if it is "yes" or zero. Tag content can be inner tags or text but not both. Text is encoded as a single string in the same way as a string attribute value.
